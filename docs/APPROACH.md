## 1. Problem Decomposition

The primary challenge is transforming ambiguous hiring conversations into grounded SHL assessment recommendations while maintaining conversational continuity, deterministic behavior, and zero hallucinations in a stateless API. The four required behaviors are: **clarify** missing requirements, **recommend** a shortlist, **refine** it based on user feedback, and **compare** specific assessments side by side. Every recommended assessment, URL, and metadata field must be traceable to the scraped SHL catalog — nothing is generated by the model.

---

## 2. Architecture and Design Rationale

The pipeline separates reasoning from generation deliberately. The first LLM call extracts structured hiring intent from the conversation. The second generates a grounded consultant response using only the catalog entries retrieved in between. This two-call separation reduces prompt complexity, keeps each call narrowly scoped, and prevents the generator from making routing or policy decisions — concerns that are better handled deterministically. This architecture also keeps the system modular, allowing retrieval, routing, and generation to evolve independently without affecting the remaining pipeline.

```
User
 ↓
Conversation Understanding  (LLM #1 — extracts hiring intent)
 ↓
Intent Routing              (deterministic rule-based)
 ↓
Hybrid Retrieval            (BM25 + dense semantic search)
 ↓
Metadata Filtering          (technology and capability reranking)
 ↓
Grounded Response Generation (LLM #2 — catalog-grounded only)
 ↓
Validation                  (catalog name verification)
 ↓
API Response                (URL and test type injected from catalog)
```

Each stage has a single responsibility, allowing retrieval, routing, and generation to evolve independently while keeping the overall pipeline simple to test and maintain. The router handles five intent types — refuse off-topic requests, clarify missing requirements, recommend, refine, and compare — based entirely on structured state flags, not on LLM judgment.

---

## 3. Design Trade-offs

**Hybrid retrieval over dense-only search.** Lexical search performs reliably for exact assessment names, while semantic search handles vague hiring descriptions. Combining both with Reciprocal Rank Fusion produces a ranked list that is robust to both precise and conversational queries.

**Deterministic routing over agentic flow.** An LLM could decide when to clarify versus recommend, but this would introduce non-determinism and complicate testing. Instead, a rule-based router acts on structured state extracted by LLM #1, guaranteeing predictable execution paths and making each stage independently testable.

**Stateless API over server-side sessions.** A stateless design aligns naturally with REST APIs, avoids distributed session management, simplifies horizontal scaling, and allows every response to be reproduced solely from the supplied conversation history. Rebuilding conversation state on every turn from the message history adds modest latency in exchange for these guarantees.

**Precision over recall in recommendations.** Metadata reranking penalizes assessments for unrelated technologies, which can reduce the number of recommendations returned. This was a deliberate trade-off: a smaller, accurate shortlist is more useful to a hiring manager than a padded list with weak matches.

---

## 3b. Reliability by Design

The system intentionally combines probabilistic and deterministic components. LLMs are used only where natural language understanding or generation is required, while routing, retrieval, validation, URL construction, and response assembly remain fully deterministic. This separation improves reproducibility, simplifies testing, and minimizes hallucination risk without sacrificing conversational flexibility.

---

## 4. Retrieval Design

Retrieval combines BM25 lexical search with FAISS-backed dense embeddings, fused using Reciprocal Rank Fusion. Neither strategy is sufficient on its own: lexical retrieval struggles with paraphrased hiring requests, while dense retrieval may surface semantically similar but technologically incorrect assessments. Hybrid retrieval balances both precision and recall. Lexical search anchors results on exact assessment names; dense retrieval captures hiring intent such as "backend engineer" or "leadership role" that would not surface through keyword matching alone. Dense retrieval improves semantic coverage, whereas lexical retrieval preserves precision for exact assessment names; combining both reduces the weaknesses of either approach alone.

After fusion, a metadata reranking stage applies deterministic adjustments based on the extracted hiring state. Direct technology matches in the assessment name receive strong positive weight; assessments focused on a different programming language or domain receive a penalty that removes them from contention before they reach the generator. Personality instruments are deterministically promoted when the hiring context explicitly requires personality assessment, regardless of their retrieval rank. This reranking step is what prevents the LLM from ever seeing irrelevant candidates.

---

## 5. Hallucination Prevention

The architecture prevents hallucinations through four independent safeguards.

- Only assessments retrieved and ranked by the search layer are exposed to the generator. The LLM cannot recommend anything outside this grounded list.
- Any assessment name in the generator output that does not exactly match the catalog is silently discarded before the response is constructed.
- URLs and test type codes are injected from the catalog at response-build time. The model never produces a URL.
- When a user requests a technology that has no dedicated catalog entry, the system clarifies rather than recommending a speculative alternative.

Every recommendation in the final API response is therefore fully traceable to the underlying SHL catalog. Consequently, the LLM is responsible only for generating natural language explanations, while every factual element of the response is enforced deterministically.

---

## 6. Conversation Design

Conversation state is rebuilt fresh on every request by passing the full message history to LLM #1. There is no server-side session store, which keeps the API horizontally scalable. The extracted state includes role, seniority, requested technologies, capability flags, and whether the user is refining a previous shortlist.

The router prioritises clarification in a fixed order — role first, then seniority, then specific skills — but delegates the decision of when enough information has been gathered to the LLM, which signals readiness through a structured flag. This keeps conversations concise by avoiding unnecessary clarification turns while still collecting enough information for relevant recommendations. For comparison requests, a catalog resolver checks all mentioned assessment names before calling the generator; if any name is unresolvable, a deterministic short-circuit reply is returned without an LLM call.

## 7. Evaluation

The evaluation methodology is fully implemented and measures retrieval quality, conversational behavior, and response correctness across three dimensions: functional correctness through unit and integration testing, retrieval quality using Recall@10 on the provided public conversation traces, and behavioral robustness using clarification, refinement, comparison, refusal, and hallucination probes.

The evaluation harness replays all ten public traces against the live API and computes Recall@10, schema violations, and hallucinated URL counts per trace. Due to provider rate limits, the complete benchmark could not be executed during the submission window. Zero hallucinated URLs were recorded — the catalog-injection architecture prevents this class of error structurally. The unit test suite covers 375 cases across the agent, retrieval, and API layers.

## 8. Engineering Challenges

**Technology leakage in retrieval.** Dense retrieval occasionally surfaced assessments for related programming languages — a Java assessment ranking highly for a Python query, for example. The root cause is that semantically similar technology descriptions produce similar embedding vectors. The solution was to introduce deterministic metadata filtering after fusion: assessments whose primary focus is a different known catalog technology are penalised and removed before the generator receives candidates. Recommendation precision improved substantially as a result.

**Incorrect conversation termination.** The generator would occasionally set end-of-conversation to true when responding to a legal or compliance question, even though the conversation was not complete. The root cause was that the generator had no awareness of the current route type. The fix was to enforce the rule deterministically in the response construction layer: end-of-conversation can only be true on recommendation and comparison routes, never on clarification or refusal routes.

**Evaluation interrupted by provider rate limits.** Running ten multi-turn conversation traces sequentially exhausted the free-tier daily token limit. The fix was to add exponential backoff and retry logic in the evaluation harness, and to make the LLM provider configurable so evaluation can be redirected to a provider with a higher quota.
